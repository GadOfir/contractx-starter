Docs

How it actually works

The technical spec. CONTRACTX is a framework, a folder structure, a skill, and a protocol — all working together.

Concepts

The hierarchy — single file vs. CONTRACTX/ folder

Every directory that becomes a durable boundary gets a CONTRACTX.md. Small directories (≤5 files, ≤2 concerns) get a single CONTRACTX.md. Large directories (10+ files, 3+ concerns) get a CONTRACTX/ folder with sub-contracts for each concern.

  • operators/ (2 files, 1 concern) → single CONTRACTX.md
  • services/ (16 files, 2 concerns) → CONTRACTX/ folder with registry.md + kb-subsystem.md
  • lib/ (134 files, 6 concerns) → CONTRACTX/ folder with dispatch.md, auth.md, policy-cascade.md, vault-secrets.md, tracing.md, config.md

The entry-point CONTRACTX.md always exists and references the CONTRACTX/ folder when one exists. The walk protocol never changes — the agent just reads more targeted files instead of one giant one.

Walk-before-edit protocol

The agent MUST walk the CONTRACTX chain before touching any file. Starting from root, read every CONTRACTX.md along the path to the target. No memory reliance. Re-read in every session.

Root first

Root CONTRACTX.md = project-wide rules + top-level child index. Always step 1.

Parent to child

lib/CONTRACTX.md → lib/handlers/CONTRACTX.md → lib/handlers/skill/CONTRACTX.md. Each adds precision.

Proximity wins

Closer doc controls local details. No child weakens parent. Conflicts resolve to the closer doc.

Folder-aware

If CONTRACTX.md references a CONTRACTX/ folder, read _index.md + relevant sub-contracts.

Post-edit update pass

After every meaningful change, the agent runs a CONTRACTX pass: update the nearest owning CONTRACTX.md, refresh child indexes, remove stale text. Small edits may skip content changes, but the pass still happens.

What triggers an update: purpose/scope/ownership changes, new invariants, workflow changes, new inputs/outputs/permissions, CONTRACTX.md creation/deletion/move/rename. Do NOT update for trivial changes — the pass confirms "no update needed" without writing.

Gap tracking (CX-NNN)

CONTRACTX-level gaps use the CX-NNN format in CONTRACTX/gaps/. Same lifecycle as domain-level G-NNN: OPEN → SHIPPED → RESOLVED. Each gap entry records the contradiction, the affected CONTRACTX.md files, the fix commit, and resolution date.

Gaps never get deleted — moved from open/ to resolved/. The audit trail is permanent. If a CONTRACTX gap reveals a domain-level contradiction, it escalates to docs/MASTER_CONTRACT.md as a G-NNN.

The contractx skill

A SKILL.md file in .claude/skills/contractx/ that gives the agent 8 workflows: init (interview + build tree), walk (traverse chain for a path), update (post-edit pass), audit (exhaustive chain verification), gap (file CX-NNN), resolve (close CX-NNN), grow (create child when new boundary emerges), cross-cut (create cross-cutting contract).

Hard rules baked into the skill: always read root first, walk before edit, gap findings go to CONTRACTX/gaps/ not inline, update parent index when child changes, no child weakens parent, audit is exhaustive. The skill IS the protocol.

Cross-cutting contracts

Some contracts span multiple directories. HMAC auth touches both lib/auth.py and web-ui/main_server.py. The ClientRPC envelope touches lib/dispatch_http.py and lib/client_rpc.py. These go in CONTRACTX/cross-cutting/ and are referenced from each affected directory's CONTRACTX.md.

One contract, many references. When the HMAC version bumps, update one file, and every directory CONTRACTX.md that links to it stays current.

FAQ
How is this different from regular docs?

Regular docs are written once and rot. CONTRACTX is maintained by the agent during every edit. The post-edit pass keeps contracts current. The walk-before-edit protocol means the agent actually reads them. And the word "contract" triggers more careful AI behavior than "documentation."

What do I need to run this?

Git and any AI agent that supports markdown instruction files (Claude Code, Codex, OpenCode, Cursor). That's it. No npm, no Python, no Docker. Clone the starter, run your agent, ask it to interview you.

Can I use this with an existing project?

Yes. That's the primary use case. Clone contractx-starter into your existing repo, run /contractx init, and answer the interview questions. The agent builds the CONTRACTX tree from your existing code structure and conventions.

Does CONTRACTX replace my existing contract docs?

No. CONTRACTX is code-level contracts (per-directory rules for agents editing code). Domain-level contracts (AUDIT_CONTRACT, CASCADE_CONTRACT, etc.) stay where they are. The two systems complement each other — CONTRACTX gaps can escalate to domain-level gaps when they reveal design contradictions.

What if my project has 200 directories?

CONTRACTX only creates child files at durable boundaries — directories with their own purpose, rules, and responsibilities. Most directories are simple containers and don't get a CONTRACTX.md. A 200-directory project might have 10-15 CONTRACTX.md files. The threshold is built into the skill.

Can the agent really maintain this automatically?

Yes — if you enforce the protocol. The contractx skill bakes in the rules: walk before edit, update after edit, never weaken parent rules. The audit workflow verifies compliance. After 2-3 sessions of enforcing it, the agent internalizes the pattern. The skill IS the enforcement.

What happens if the agent ignores the contracts?

Run /contractx audit. It detects drift: stale indexes, missing children, rules the agent violated, invariants that changed without updating the contract. Every finding becomes a CX-NNN gap with a fix path. The audit is your safety net.
